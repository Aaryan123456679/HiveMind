# Pending

- **btree SaveRoot / WAL-replay gap (Phase 2 follow-up)**: `engine/btree`'s `SaveRoot` is a manual, out-of-band checkpoint, not auto-invoked on every `Insert`; `RecoverFromWAL` deliberately no-ops on btree WAL records. A real crash between an `Insert` and the next `SaveRoot` would silently drop that insert from the recovered btree, even though its WAL record was durably applied. Pre-existing design from issue #2 (task-1.2), surfaced and transparently documented (not fixed) during verification of task-1.5.2 (`.cdr/runs/2026-07-04/064-verification/`). Recommendation: either wire btree WAL records into real replay-based reconstruction, or add automatic/periodic `SaveRoot` checkpointing, plus an explicit test exercising the currently-uncheckpointed window.

- **Node-latch registry has no eviction (Phase 2a follow-up)**: `engine/btree/latch.go`'s `NodeStore`-owned `map[uint64]*nodeLatch` registry (added in task-2a.4.1, issue #9) grows unboundedly with the number of distinct node IDs ever locked/versioned; entries are never evicted. Undisclosed as a known limitation in `latch.go`'s own docs, though precedented elsewhere in the codebase (e.g. `engine/mvcc`'s `commitLocks`). Surfaced as a non-blocking finding during verification of task-2a.4.1 (`.cdr/runs/2026-07-05/017-verification/`). Recommendation: revisit once 2a.4.2-2a.4.5 land and real workload/memory pressure is known; consider an LRU-bounded or reference-counted registry if it becomes an issue.

- **node.go stale doc comments on the on-disk Version field (Phase 2a follow-up)**: `engine/btree/node.go`'s doc comments for the on-disk `LeafNode.Version`/`InternalNode.Version` field still describe "actual CAS/atomic version-bump logic" as future work, but task-2a.4.1 (issue #9) resolved that design question by building a separate, in-memory-only version counter in `latch.go` instead of reusing the on-disk field. Surfaced as a non-blocking finding during verification of task-2a.4.1 (`.cdr/runs/2026-07-05/017-verification/`). Recommendation: update `node.go`'s comments to point at `latch.go`'s `NodeStore.Version`/`WriteNode` and clarify the on-disk field is not used for in-process concurrency control.

- **No retry cap on TryLock restart-from-root loop (Phase 2a follow-up)**: `engine/btree/insert.go`'s crabbing insert path (task-2a.4.2, issue #9) converts hand-over-hand latch acquisition to `NodeStore.TryLock`, releasing all held latches and restarting from the root (with jittered backoff) on any miss, to eliminate a genuine deadlock found in round 3 of that subtask. The restart loop itself has no maximum-attempt cap: under adversarial scheduling this is a theoretical (never observed, across ~40 min of stress testing) livelock risk, though it carries no data-loss-via-give-up risk since the loop never gives up. Surfaced as a non-blocking finding during the 4th (final) verification pass of task-2a.4.2 (`.cdr/runs/2026-07-06/002-verification/`). Recommendation: consider an optional attempt counter/metric (not a hard cap) to make pathological retry storms observable; revisit if 2a.4.3 (delete) or 2a.4.5 (mixed workload) — which should reuse this same TryLock+restart pattern — surface any real-world livelock signal.
