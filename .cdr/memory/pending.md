# Pending

- **btree SaveRoot / WAL-replay gap (Phase 2 follow-up)**: `engine/btree`'s `SaveRoot` is a manual, out-of-band checkpoint, not auto-invoked on every `Insert`; `RecoverFromWAL` deliberately no-ops on btree WAL records. A real crash between an `Insert` and the next `SaveRoot` would silently drop that insert from the recovered btree, even though its WAL record was durably applied. Pre-existing design from issue #2 (task-1.2), surfaced and transparently documented (not fixed) during verification of task-1.5.2 (`.cdr/runs/2026-07-04/064-verification/`). Recommendation: either wire btree WAL records into real replay-based reconstruction, or add automatic/periodic `SaveRoot` checkpointing, plus an explicit test exercising the currently-uncheckpointed window.

- **Node-latch registry has no eviction (Phase 2a follow-up)**: `engine/btree/latch.go`'s `NodeStore`-owned `map[uint64]*nodeLatch` registry (added in task-2a.4.1, issue #9) grows unboundedly with the number of distinct node IDs ever locked/versioned; entries are never evicted. Undisclosed as a known limitation in `latch.go`'s own docs, though precedented elsewhere in the codebase (e.g. `engine/mvcc`'s `commitLocks`). Surfaced as a non-blocking finding during verification of task-2a.4.1 (`.cdr/runs/2026-07-05/017-verification/`). Recommendation: revisit once 2a.4.2-2a.4.5 land and real workload/memory pressure is known; consider an LRU-bounded or reference-counted registry if it becomes an issue.

- **FileGuard registry has no eviction (Phase 2b follow-up)**: `engine/split/guard.go`'s `FileGuard`-owned `map[uint64]*fileSplitState` registry (added in task-2b.1.2, issue #10) grows unboundedly with the number of distinct fileIDs ever guarded, entries never evicted â€” same deliberately-deferred growth characteristic as `engine/btree/latch.go`'s `NodeStore.latches` noted above; revisit together if/when that one is addressed.

- **Abandoned SPLITTING record has no automatic recovery (Phase 2b follow-up)**: `engine/split/orchestrate.go`'s `Orchestrator` (task-2b.1.3, issue #10) transitions a catalog record to `StatusSplitting` in `BeginSplit` and relies on the split holder eventually calling `EndSplit`/`AbortSplit` to transition back out and release `FileGuard`. If a split holder crashes between `BeginSplit` and `EndSplit`/`AbortSplit`, the record is left permanently `StatusSplitting` (and, if the process itself survives, `FileGuard`'s in-memory flag is left permanently held) with no timeout or automatic reversion -- `BeginSplit` will refuse (`ErrAlreadySplitting`) all future split attempts for that fileID forever. Explicitly and honestly disclosed in `orchestrate.go`'s doc comments and the 2b.1.3 commit message as deliberately out of scope (no split executor exists yet; issue #12's "Atomic split-transaction execution" is expected to own the real commit path that would make crash-injection/recovery testing meaningful). Surfaced as a non-blocking finding during verification of task-2b.1.3 (`.cdr/runs/2026-07-07/008-verification/`). Recommendation: issue #12 (or a dedicated follow-up) should decide on a recovery story -- e.g. a lease/heartbeat-based timeout on `StatusSplitting`, or an explicit crash-recovery repair pass -- before any real split executor is wired to `BeginSplit`/`AdmitWrite` in production.

- **Canonical `newFileIDs` ordering contract needed before 2b.3.6 (Phase 2b follow-up)**: `engine/split/execute.go`'s `ExecuteSplitRedirectStub` (task-2b.3.2, issue #12) writes a deterministic redirect-stub content marker, but that determinism depends entirely on the caller supplying `newFileIDs` in a stable order -- the function does no internal sorting or canonicalization of its input. 2b.3.1's `ExecuteSplitAllocateAndWrite` returns its new-fileID mapping as a plain Go `map[string]uint64` (NewPath -> fileID), which has no defined iteration order. Surfaced as a non-blocking finding during verification of task-2b.3.2 (`.cdr/runs/2026-07-07/018-verification/`). Recommendation: before 2b.3.6 wires 2b.3.1's map output into `ExecuteSplitRedirectStub`, pin a canonical ordering contract (e.g. sort by `NewPath` lexicographically, or by allocated fileID) at the integration seam, so stub-content determinism doesn't silently depend on Go's unspecified map iteration order.

- **Raw topic-path strings used directly as B+Tree keys, no normalization/namespace layer (Phase 2b follow-up)**: `engine/split/execute.go`'s `ExecuteSplitBtreeInsert` (task-2b.3.3, issue #12) inserts new and old topic paths into `engine/btree`'s `*btree.Tree` using the raw path string as the key, with no normalization, canonicalization, or namespace/prefix layer. This works for the current split-execution primitives but may need reconciliation once a real topic-path indexing scheme is designed. Surfaced as a non-blocking finding during verification of task-2b.3.3 (`.cdr/runs/2026-07-07/020-verification/`). Recommendation: whichever future subtask/issue designs the canonical topic-path indexing convention should account for this direct-raw-string-as-key usage and decide whether normalization needs to be retrofitted here.
