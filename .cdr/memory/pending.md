# Pending

- **btree SaveRoot / WAL-replay gap (Phase 2 follow-up)**: `engine/btree`'s `SaveRoot` is a manual, out-of-band checkpoint, not auto-invoked on every `Insert`; `RecoverFromWAL` deliberately no-ops on btree WAL records. A real crash between an `Insert` and the next `SaveRoot` would silently drop that insert from the recovered btree, even though its WAL record was durably applied. Pre-existing design from issue #2 (task-1.2), surfaced and transparently documented (not fixed) during verification of task-1.5.2 (`.cdr/runs/2026-07-04/064-verification/`). Recommendation: either wire btree WAL records into real replay-based reconstruction, or add automatic/periodic `SaveRoot` checkpointing, plus an explicit test exercising the currently-uncheckpointed window.

- **Node-latch registry has no eviction (Phase 2a follow-up)**: `engine/btree/latch.go`'s `NodeStore`-owned `map[uint64]*nodeLatch` registry (added in task-2a.4.1, issue #9) grows unboundedly with the number of distinct node IDs ever locked/versioned; entries are never evicted. Undisclosed as a known limitation in `latch.go`'s own docs, though precedented elsewhere in the codebase (e.g. `engine/mvcc`'s `commitLocks`). Surfaced as a non-blocking finding during verification of task-2a.4.1 (`.cdr/runs/2026-07-05/017-verification/`). Recommendation: revisit once 2a.4.2-2a.4.5 land and real workload/memory pressure is known; consider an LRU-bounded or reference-counted registry if it becomes an issue.

- **FileGuard registry has no eviction (Phase 2b follow-up)**: `engine/split/guard.go`'s `FileGuard`-owned `map[uint64]*fileSplitState` registry (added in task-2b.1.2, issue #10) grows unboundedly with the number of distinct fileIDs ever guarded, entries never evicted — same deliberately-deferred growth characteristic as `engine/btree/latch.go`'s `NodeStore.latches` noted above; revisit together if/when that one is addressed.

- **Abandoned SPLITTING record has no automatic recovery (Phase 2b follow-up)**: `engine/split/orchestrate.go`'s `Orchestrator` (task-2b.1.3, issue #10) transitions a catalog record to `StatusSplitting` in `BeginSplit` and relies on the split holder eventually calling `EndSplit`/`AbortSplit` to transition back out and release `FileGuard`. If a split holder crashes between `BeginSplit` and `EndSplit`/`AbortSplit`, the record is left permanently `StatusSplitting` (and, if the process itself survives, `FileGuard`'s in-memory flag is left permanently held) with no timeout or automatic reversion -- `BeginSplit` will refuse (`ErrAlreadySplitting`) all future split attempts for that fileID forever. Explicitly and honestly disclosed in `orchestrate.go`'s doc comments and the 2b.1.3 commit message as deliberately out of scope (no split executor exists yet; issue #12's "Atomic split-transaction execution" is expected to own the real commit path that would make crash-injection/recovery testing meaningful). Surfaced as a non-blocking finding during verification of task-2b.1.3 (`.cdr/runs/2026-07-07/008-verification/`). Recommendation: issue #12 (or a dedicated follow-up) should decide on a recovery story -- e.g. a lease/heartbeat-based timeout on `StatusSplitting`, or an explicit crash-recovery repair pass -- before any real split executor is wired to `BeginSplit`/`AdmitWrite` in production. **Update (task-2b.3.6, issue #12, verified PASS_WITH_COMMENTS, `.cdr/runs/2026-07-07/027-verification/`): narrowed, not eliminated.** `ExecuteSplitAtomic`'s single `wal.AppendAndApply` closure is the first point `Status` actually leaves `StatusSplitting` on the real (atomic) code path, so for that path the exposure window shrinks from "any time between `BeginSplit` and `EndSplit`/`AbortSplit`" down to "strictly before `ExecuteSplitAtomic`'s own WAL record fsync" -- a materially smaller window, but a crash before that fsync still leaves the record stuck `StatusSplitting` with no automatic recovery. The underlying recovery-story gap (lease/heartbeat timeout or explicit repair pass) remains genuinely open, now scoped to that narrower pre-fsync window.

- **[RESOLVED by task-2b.3.6, issue #12] `engine/graph` edge-append records have no crash-recovery replay path (Phase 2b follow-up)**: `engine/graph`'s new `EdgeAppender.AppendEdge` (task-2b.3.4, issue #12) durably fsyncs and CRC-checks each `SPLIT_SIBLING`/`REDIRECT` edge record via `engine/wal`'s low-level segment writer, but does **not** go through `engine/wal`'s `TypedRecord`/`Replay` recovery layer the way catalog/btree records do. This means edge-append records are byte-durable on disk but are currently invisible to any crash-recovery reconstruction path. Surfaced as the most consequential non-blocking finding during verification of task-2b.3.4 (`.cdr/runs/2026-07-07/022-verification/`). **Update (task-2b.3.5, issue #12, verified PASS_WITH_COMMENTS, `.cdr/runs/2026-07-07/024-verification/`): deliberately not resolved by 2b.3.5.** 2b.3.5's scope was edge *content* (wiring `ExecuteSplitGraphEdges` to append the correct SPLIT_SIBLING/REDIRECT edges) and it correctly stayed out of edge *durability-under-crash*, which was never in its acceptance criteria. **Update (task-2b.3.6, issue #12, verified PASS_WITH_COMMENTS, `.cdr/runs/2026-07-07/027-verification/`): RESOLVED.** `ExecuteSplitAtomic`'s new `wal.RecordSplitCommit` record wraps the graph edge appends inside the same single `wal.AppendAndApply` closure as the catalog `Put` and btree insert, and `RecoverSplitCommits` replays them idempotently via `graph.EdgeAppender.AppendEdgeIfAbsent` (dedupes on byte-identical `(Source,Target,Type)`). Edge-append durability-under-crash is now integrated into the same recovery path as catalog/btree. Non-blocking caveat carried forward from 2b.3.6's own verification: the crash-injection subtests exercising this replay path use the same in-memory objects rather than freshly reconstructed ones (see the dedicated item below), so this is proven correct for idempotent re-application onto partially-mutated state, not yet for a fully faithful process-restart simulation.
