# task-3.2.3 — `engine/split/proposer_grpc.go` real gRPC-backed `SplitProposer` client (issue #16, Epic Phase 3)

## Summary

Subtask 3.2.3 ("wire `engine/split/` to a real gRPC-backed `SplitProposer` implementation calling the Python agent service's `ProposeSplit` RPC") is complete and independently verified PASS with zero findings. This is the third increment of issue #16's five-part gRPC rollout (3.2.1-3.2.5), consuming the shared `hivemind.proto` contract (3.2.1) and the real server handlers (3.2.2) to add a genuine client-side transport adapter. `engine/split/proposer_mock.go` (task-2b.2.2's deterministic mock) and `engine/rpc/server.go` are both untouched — purely additive.

## Features

- `engine/split/proposer_grpc.go`: new `GRPCSplitProposer`, a thin client-side adapter implementing the existing `split.SplitProposer` interface via `hivemindv1.HiveMindClient.ProposeSplit`. Marshals `fileContent` into a `ProposeSplitRequest`; unmarshals the response back into `SplitPlan`/`SplitFileProposal`/`SectionRange`, with a documented, sound `int64`-wire-to-`int`-internal offset conversion. Supports an optional per-call `context.WithTimeout`. Any gRPC error (transport failure, non-OK status including `codes.Unimplemented` — the real server's current state since 3.2.2 left `ProposeSplit` as the generated Unimplemented stub — or deadline exceeded) is wrapped and returned as a non-nil error with a zero `SplitPlan`, never a panic.
- `engine/split/proposer_grpc_test.go`: `TestGRPCSplitProposer`, a genuine wire round-trip test via `bufconn.Listener` + a real `grpc.Server`/`grpc.NewClient` (not a mocked-out Go interface). Four subtests: `happy_path` (with a channel-based cross-goroutine assertion that the fixture server genuinely received the marshaled request content), `empty_response` (nil-safe proto accessor handling, yields empty non-nil `SplitPlan{}`), `server_error` (`codes.InvalidArgument` propagates end-to-end), and a wall-clock-timed `timeout` test (20ms client timeout vs. 300ms server delay, asserting elapsed time is well under 300ms — proving the timeout is actually enforced, not just an error-code check).
- `docs/LLD/rpc.md`: status paragraph updated to accurately reflect 3.2.2/3.2.3 completion, without overclaiming server-side `ProposeSplit`/LLM logic completion (that remains issue #18's scope).

## Impact

Acceptable for this subtask's scope; flagged only for a future interface revision if per-call deadlines become necessary.
