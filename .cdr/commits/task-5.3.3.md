# task-5.3.3: Aggregate per-stage latency and $/1000-query cost across benchmark arms (issue #28)

## Summary

**Problem:** Issue #28 (milestone #7 cross-arm benchmark comparison, Phase 5) subtask 5.3.3 calls for rolling up per-pipeline-stage latency/cost data -- sourced from the `agents/llm/`/`engine/rpc/` interceptors referenced in task-3.2.4 -- into a single $/1000-query figure per benchmark arm. Architecture discovery during implementation found that issue text's premise does not fully hold: task-3.2.4 only ever shipped the Go-side gRPC interceptor (`engine/rpc/interceptor.go`), whose own header comment explicitly defers a Python-side LLM interceptor (`agents/llm/interceptor.py` or similar) as out of scope, and no later subtask built one. `agents/llm/` has no interceptor, cost, or token-usage capture code anywhere, and no OpenRouter/Gemini pricing table exists anywhere in the repo. The three baseline arms (vector-RAG, vector-RAG+rerank, GraphRAG-lite) plus the HiveMind arm also use differing per-arm pipeline stage sets, so a fixed stage enum would not generalize.

**Solution:** Added `agents/eval/cost_latency.py`, defining a structured per-stage record shape (`StageRecord`) modeled on the Go-side `RPCMetric` fields (call/stage name, duration, optional cost) that a real interceptor would need to emit, plus `aggregate_by_stage()` (per-arm, per-stage exact totals/means) and `rollup_cost_per_1000_queries()` (per-arm $/1000-query, normalized by distinct `query_id` count, correctly scoped per-arm so duplicate query ids across different arms don't leak into each other's counts). `resolve_cost_usd()` implements an explicit three-branch cost policy: trust an explicit `cost_usd` if present, default Ollama-provider records to $0.0, and raise `ValueError` for any other provider missing `cost_usd` rather than silently inventing a per-token rate. The scope deviation (the Python-side interceptor log format this module is designed to consume doesn't yet exist) is disclosed in the module docstring and commit message rather than worked around silently, consistent with the 5.2.4 precedent for differing per-arm stage sets. `agents/eval/test_cost_latency_aggregation.py` adds 17 hand-verified fixture-based tests covering both functions, `resolve_cost_usd`'s three branches, and edge cases (empty input, all-Ollama arm, multi-stage single query, duplicate query_id across differing arms).

Files touched: `agents/eval/cost_latency.py` (new), `agents/eval/test_cost_latency_aggregation.py` (new). No existing `agents/eval/` file touched, no new dependency (`agents/pyproject.toml` diff empty), no LLM/network calls introduced. `agents/eval/` full suite: 122 passed.

## Impact

`agents/eval/` now has a reusable, standalone cost/latency rollup module that a future benchmark harness (5.3.4) can feed real per-run interceptor-style records into, without waiting on the still-unbuilt Python-side LLM interceptor. It is not yet wired into any harness/pipeline call site -- expected at this stage, since 5.3.4 is the intended integration point (non-blocking finding F2 below).

**Important forward-looking gap for 5.3.4 (non-blocking here, but must be addressed there -- finding F1):** `resolve_cost_usd()`'s ValueError-on-missing-cost behavior for any non-Ollama-provider record is a real forward blocker for 5.3.4. Because no OpenRouter/Gemini client currently emits `cost_usd` anywhere in `agents/llm/`, and no pricing/rate table exists in the repo, a real (non-fixture) paid-arm benchmark run will raise `ValueError` on its very first non-Ollama record. **5.3.4's implementer must read and explicitly address this** before scoping that work -- e.g. by having 5.3.4 supply an estimated per-token pricing table, or by having the LLM clients start recording actual `cost_usd`. This is tracked in `.cdr/index/regression.jsonl` (entry at commit `50c2b06`, run `175-verification`, severity medium, status open, `blocking: false`).

**F2 (non-blocking, expected at this stage, not a defect):** `cost_latency.py` isn't yet wired into any harness/pipeline call site. This is by design -- 5.3.4 is where the integration and call-site ergonomics (e.g. whether callers should catch the `ValueError`, log-and-skip, or fail the run) get decided.
