# task-2b.1.3 — SPLITTING orchestration: mark, queue, isolate (subtask 3 of 3, issue #10)

## Summary
Third and final subtask under GitHub issue #10 ("[2b] Split trigger + per-file CAS guard", Epic Phase 2b: Auto-split). Adds `engine/split/orchestrate.go`: an `Orchestrator` that composes 2b.1.2's `FileGuard` CAS guard with `CatalogRecord.Status` transitions, so that beginning a split (a) is admitted to exactly one caller via the guard, (b) durably marks the catalog record `StatusSplitting` (WAL-before-apply), (c) causes new writer requests to be refused/queued rather than silently applied while splitting, and (d) leaves existing/new MVCC reader snapshots completely unaffected for the duration of the in-flight split. This closes out issue #10 in its entirety — all three subtasks (2b.1.1 threshold detection, 2b.1.2 CAS guard, 2b.1.3 this orchestration layer) are now implemented and independently verified.

## Features
- `engine/split.Orchestrator`: `BeginSplit` (guard.TryAcquire → transitionStatus to `StatusSplitting`, guard released on any post-win failure), `EndSplit`/`AbortSplit` (validated exit transition — back to `StatusActive` on abort, forward on success — with unconditional guard release via deferred cleanup, including on the `ErrNotSplitting` refusal path), and `AdmitWrite` (write-admission gate: refuses new writers with a sentinel error while a file is `StatusSplitting`, rather than silently applying them).
- Composes strictly on top of 2b.1.2's already-verified `FileGuard.TryAcquire` (a genuine atomic CompareAndSwap) — no status mutation is ever reachable except via the CAS-serialized path, so `transitionStatus`'s read-then-write is safe by construction, not by convention.
- Status transitions follow the repo's established WAL-before-apply idiom (`wal.NewCatalogPutRecord` + `wal.AppendAndApply`, then `cat.Put`), matching `ContentStore.Append`'s precedent.
- Specific, well-documented sentinel errors per call site (`ErrAlreadySplitting`, `ErrNotSplitting`, `ErrSplitInProgress`, `ErrUnexpectedStatus`) rather than one generic error, including correct handling of a stuck/crashed-holder `StatusSplitting` record left behind (distinct from a simple sequential double-call).

## Impact
- **Issue #10 is now fully closed**: all 3 subtasks (2b.1.1, 2b.1.2, 2b.1.3) implemented and independently verified.
- Reader isolation was **proven, not just asserted**: `reader_snapshot_unaffected_by_splitting` takes a real `mvcc.NewSnapshot` before `BeginSplit`, reads it during `StatusSplitting`, and asserts byte-identical content against the committed version both during and after the split window. This is backed by an independent structural check — `engine/mvcc/read.go` and `write.go` have zero references to `CatalogRecord.Status` anywhere; `NewSnapshot` pins only `CurrentVersion`, so Status is orthogonal to MVCC by construction, not by luck.
- Concurrency was proven under genuine contention: `concurrent_writers_and_split_race` runs 32 goroutines truly racing `AdmitWrite` against a concurrent `BeginSplit`/`AbortSplit` under `-race`, asserting both `admitted > 0` and `refused > 0` with no unexpected errors.
- **Honestly-disclosed gap (non-blocking, tracked)**: if a split holder crashes between `BeginSplit` and `EndSplit`/`AbortSplit`, the catalog record is left permanently in `StatusSplitting` (and, if the process survives, `FileGuard`'s in-memory flag remains held) — there is no timeout or automatic recovery, so `BeginSplit` will refuse all future split attempts on that fileID forever. This is disclosed in `orchestrate.go`'s doc comments and explicitly deferred because no split executor exists yet (issue #12 owns real split execution, which is what would make crash-injection/recovery testing meaningful). Per the verifier's recommendation, this gap has been recorded in `.cdr/memory/pending.md` so it is discoverable by issue #12's planning without requiring this subtask's implementation notes.
- No regression: `engine/split/guard.go` and `engine/split/trigger.go` are untouched (byte-identical to the prior verified commit `bed7edd`); full `go test ./catalog/...`, `./btree/...`, `./mvcc/...` all green.
