# task-3.1.2 — Append-only per-node edge log writer (issue #15, Epic Phase 3)

## Summary
Subtask 3.1.2 of GitHub issue #15 (Epic Phase 3) complete and independently verified (PASS_WITH_COMMENTS). Adds `engine/graph`'s durable, low-latency append path for new graph edges (`ENTITY_COOCCUR`, `LLM_ASSERTED`, `SPLIT_SIBLING`, `REDIRECT`): a per-source-fileID `EdgeLog` that lands writes ahead of periodic compaction into 3.1.1's CSR array (`graph.dat`). Concurrent writers touching different source fileIDs (e.g. parallel ingestion workers) no longer contend on a single shared lock. `docs/LLD/graph.md` updated. Issue #15 stays open: 4 more subtasks remain (3.1.3-3.1.6).

## Features
- `engine/graph/edgelog.go`'s `EdgeLog`: append-only log manager keyed by source fileID, one `wal.Writer` per distinct fileID, backed by a per-node subdirectory (`<root>/<fileID>/wal-N.log`).
- Lazy writer creation via correct double-checked locking (`sync.RWMutex`: RLock fast path, then Lock + re-check before creating and registering a new `wal.Writer`) — closes the classic DCL race where two goroutines both miss the fast path and both create separate writers.
- Reuses `csr.go`'s `CSREdge` type (Target, Type, Weight, LastUpdated) for the append-log record shape, with no CSR offset/index-array baggage, directly setting up 3.1.3's compaction to merge without a translation layer.
- Every append fsyncs, matching `edge_append.go`'s existing `EdgeAppender` durability guarantee; no buffering introduced.
- Scope kept intentionally narrow: `edge_append.go`'s split-specific `EdgeAppender` (SPLIT_SIBLING/REDIRECT during split execution, closed out under task-2b.3.6/issue #12) left byte-identical and untouched — 3.1.2 is a separate, general-purpose per-node append path per issue #15's scoping.
- 5 new tests (`edgelog_test.go`): basic append/read, invalid edge type, fsync-durability across Close+reopen, 50-goroutine/50-distinct-fileID concurrent correctness (per-node count/order/target isolation, catching cross-node leakage), and log-close behavior.
